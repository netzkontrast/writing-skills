# Principles

How every skill in this catalog is designed. The north star: **genuinely useful** —
each skill does for the writer what the real service does, at the depth a
professional would, or it doesn't ship.

[The one rule](README.md#the-one-rule) stands above all of these: no skill writes
or rewrites the author's prose. What follows is how the *reading* side earns its keep.

## 1. Simulate the service, not the vibe

Each skill mirrors its real-world counterpart's actual process and deliverable —
a developmental editor's intake questionnaire, full read, and edit letter with
strengths first — not a generic "AI feedback" shape wearing an editor costume.
The catalog maps to services a writer could actually buy: where the trade
distinguishes two services (a manuscript assessment vs. a full developmental
edit), a skill may split along that line — guided by usefulness, never by
symmetry.

## 2. Read the book it's trying to be

Context before critique. Establish genre, tradition, era, POV convention,
audience, and the author's stated intent — then judge the manuscript against
*its own* ambitions. Don't blame the author for not achieving what they never
attempted.

## 3. Declare the lens — applied, re-scoped, set aside

Before commenting, the skill states which principles it will apply, which it is
re-scoping for this book, and which it is setting aside — so the author knows
how to read the comments.

Re-scope beats set-aside. "Show, don't tell" isn't wrong for an Austen-style
novel — it's mis-scoped: the turning points must still be dramatized, and a
high density of authorial narration is not a defect there. The lens says that
explicitly, instead of pretending the principle vanished.

**Adaptive intake:** ask upfront only what can't be inferred (a real editor's
questionnaire), declare the inferred profile at the top of the output, and
treat it as falsifiable — correct the profile and the read re-aims.

## 4. Location, evidence, stakes

Every note names where (chapter/scene), points at what's on the page, says why
it matters, and ends in an open question toward a fix the author will make.
The **any-manuscript test**: if a comment could be pasted into any book's edit
letter, it gets cut or sharpened until it can't be.

## 5. Root causes over symptom lists

Two or three ranked root diagnoses; downstream symptoms grouped under them,
never enumerated as peers. Anti-bloat structurally, not just tonally — short
because it's organized, not because it's thin.

## 6. Verdicts carry confidence

A craft-consensus flaw and a this-editor's-taste call are marked as different
things. "Not vague" also means not falsely certain.

## 7. Professional content, accessible register

The default reader is an ambitious beginner or a self-published author, not an
MFA graduate. The diagnosis is never dumbed down; the delivery explains craft
terms at first use and gatekeeps nothing. The context profile adapts the
register upward when the writer clearly doesn't need the glossary — we don't
fork skills by audience.

## 8. Standalone first — Calliope earns its keep, never gates the skill

Every skill in this repo must be **fully useful with zero Calliope**: paste or
attach text, and it works and returns its result. The Calliope (MCP) path may
make a skill *better* — reading the real book, writing findings to the Map or
marginalia — but its absence never makes a skill non-functional. A skill that
can't run without the connector doesn't belong in this repo.

So each `SKILL.md` keeps its `## With Calliope (MCP)` section **brief** — an
honest description of the enhancement, not a wiring diagram. The deep
optimization (orchestrating the actual tools) lives in Calliope's downstream
**fork** of this repo, which specializes each general skill for the product.
This repo is the clean, general upstream; the fork is where Calliope-specific
depth accrues. `writes_back` in the frontmatter is the integration signal to
that fork — it never implies the public skill needs an account to function.

## How these land in a SKILL.md

This file is canon for contributors; skills travel alone once installed, so
each `SKILL.md` embeds the working form: an adaptive-intake step, a declared
lens in the output contract, output rules that enforce #4–#6, and a brief
Calliope-enhancement section per #8 — compact, governing behavior, not pasted
in as prose.
